LVIS Ask-User-Question Card — design mockup

Inline chat-side multi-question prompt for the ask_user_question LLM tool. Component lives at src/ui/renderer/components/AskUserQuestionCard.tsx and is rendered inline at the tail of the chat stream (replaces the legacy floating FloatingQuestionPanel). Implementation reference: PR #334 (initial floating), later inlined for in-stream consistency.

2. Scenario A — Single question (1 of 1)

Most common path. The LLM emits 3 choices with recommendedIndex: 1 and altIndices: [2]. Click any choice → card submits and resolves. Free-text fallback is shown only when allowFreeText: true.

"이번 주 회의록 정리 시작할까요?"
오후 2:14
회의록 정리 어떤 형식으로 만들어 드릴까요?
❓ 질문
출력 형식을 골라주세요. 회의 길이가 30분 이상이면 요약형이 더 보기 쉽습니다.
State annotations
  • Recommend — single chip per question; recommendedIndex 1 entry max.
  • 대안 — secondary recommends; disjoint with the Recommend chip; rendered with neutral-muted tint.
  • Selected choice = solid primary fill. Pre-fix UX issue: duplicate-label choices were marked together when comparing by string — now compared by choiceIndex.
  • 20-char anchor is advisory — UI never truncates; prompt enforces upstream.

3. Scenario B — Paginated multi-question (3 questions)

Carousel through each question. Bottom-right button cycles 다음 → 검토 (last step) → 보내기 (confirm). 이전 returns to a prior step. Card state persists per request.id.

Step 1 / 3 · 회의 길이
❓ 질문 · 1 / 3

Step 2 / 3 · 형식
❓ 질문 · 2 / 3
출력 형식을 골라주세요.
Step 3 / 3 · 공유
❓ 질문 · 3 / 3
완성 후 자동 공유할까요?
Step 4 / 3 · 검토 (confirm)
❓ 질문 · 검토

Confirm row label: summaryHint if the model provided one (≤ 10 char Korean by prompt contract); otherwise truncated question at ~14 chars (truncateForLabel in component). Missing-answer rows show the answer text in amber.

5. IPC + state contract

Event Payload UI behavior
ask_user_question tool emits {id, questions[], urgent, createdAt} Card mounted at tail of askQuestions queue in ChatView.
User picks choice (single Q) respondAskUserQuestion({requestId, answers: [{choice}]}) Card resolves; LLM tool_result has the answer.
User submits multi-Q respondAskUserQuestion({requestId, answers: [{choice or freeText}, ...]}) All answers in one IPC call.
User dismisses respondAskUserQuestion({requestId, dismissed: true}) LLM tool_result: dismissed: true → fallback to defaults.

6. Open design notes

  • Single-line input only. The legacy popup used Textarea — vertical bloat made the card feel like a modal. Korean ≤ 20 chars by prompt contract.
  • No badges baked into choice.label. The model emits a flat answer string; recommend / alt placement is purely visual via recommendedIndex / altIndices. Keeps the 20-char anchor on the actual answer text.
  • Index-based selection. Duplicate choices labels are visually disambiguated by choiceIndex; comparing by string would mark every same-label chip as selected.
  • Persisted state per request.id. Re-mounts with the same id keep step + drafts; a new id resets.
  • Inline placement. Card sits at the tail of the chat stream (after the active turn's entries), not as a floating overlay. PR #334's FloatingQuestionPanel is removed; rationale: keep question-flow part of conversation history rather than a layered modal.
