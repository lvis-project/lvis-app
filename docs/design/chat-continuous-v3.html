2. Synthesis — "Continuous UX + Invisible Checkpoint"

사용자 멘탈 모델: 카톡처럼 끊김 없는 한 대화. 백엔드: rotation/summary/title chain 자동 — 사용자에게는 미니멀 inline marker 만 노출.

사용자가 보는 것 (UX layer)
  • 📅 Day separator (날짜 변경 시)
  • 📌 Inline checkpoint divider (체크포인트 발생 시 — 단, 시각적으로 미니멀)
  • 아래 dim 한 1줄 summary toast ("이전까지 다룬: ...")
  • 그 외에는 message 만 끊김 없이 stack
  • Reverse infinite scroll — 위로 올리면 이전 1일치 자동 로드
백엔드가 하는 것 (logic layer)
  • 3-tier checkpoint trigger (token / semantic / time)
  • Auto-summary at checkpoint (rolling, accumulated)
  • Title chaining (10-20자, 매 final answer 직전 incremental)
  • LLM context: system + rolling summary + recent N turns + input
  • Session DB: checkpoint chain 저장 (parent linkage)

3. Checkpoint trigger — 3-tier

Tier-1 — Hard (token, OpenCode 패턴)
  • ctx token usage ≥ 85% → 강제 checkpoint
  • 오래된 messages → summary 로 압축 (최근 cushion 40k 토큰 보존)
  • 비용: 1 LLM call (summary, model 따라 ~$0.001-0.01)
  • 안 하면? → context overflow, 응답 잘림 / 에러
Tier-2 — Semantic (topic, Warp 패턴)
  • LLM judge inline — system prompt 에 지시: "topic 이 크게 바뀌면 final answer 끝에 [checkpoint-suggested] 마커 emit"
  • 비용: 거의 0 — assistant 응답에 piggy-back, 추가 LLM call 없음
  • 마커 발견 시 → summary + new checkpoint
  • 안 하면? — 사용자가 새 주제로 갔는데 LLM 이 이전 컨텍스트 갖고 혼란
Tier-3 — Soft (time/length)
  • 24h 경과 OR 30+ messages → silent rotation
  • 비용: 0 (단순 timestamp/count 비교)
  • summary 는 if ctx ≥ 10% 일 때만 (사용자 6번 결정)
  • 자연스러운 일일 boundary, kakao day separator 와 자연 align
결정 트리 (post-turn-hook 안)
on turn_complete:
  // 1) Title chain — 매 turn 끝마다 (system prompt 지시로 inline emit, 비용 0)
  if assistant emits <title>X</title>: session.title = X.slice(0, 20)

  // 2) Tier-1: hard token check
  if ctxUsage >= 0.85:
    summary = await summarize(messages[older_part])
    rotateCheckpoint(summary)
    return

  // 3) Tier-2: semantic check (LLM 이 inline 으로 emit)
  if assistant emits [checkpoint-suggested]:
    if ctxUsage >= 0.10:
      summary = await summarize(messages[since_last_checkpoint])
    else:
      summary = null
    rotateCheckpoint(summary)
    return

  // 4) Tier-3: soft (silent, no LLM call unless ctx threshold)
  if (now - lastCheckpoint > 24h) OR (msgCount >= 30):
    if ctxUsage >= 0.10:
      summary = await summarize(messages[since_last_checkpoint])
    else:
      summary = null
    rotateCheckpoint(summary)

4. Title chaining (10-20자, 매 final answer 직전, incremental)

사용자 결정사항 정리
  • 제목 길이: 10-20자
  • 갱신 시점: 최종 답변 제출 직전 (assistant 가 final answer emit 끝나기 전)
  • 방식: 기존 title + 다음 최종답변 → 신규 title (chaining, 누적 진화)
옵션 A — 별도 LLM mini-call (post-turn-hook)
on turn_complete:
  newTitle = await llm.complete(
    `현재 제목: "${session.title}"
     마지막 답변: "${finalAnswer}"
     이 둘을 반영한 10-20자 제목:`,
    { max_tokens: 30 }
  )
  session.title = newTitle.slice(0, 20)

장점: 신뢰도 높음, 명확

단점: 매 turn 1 mini-call (~$0.0001 model 따라). 1000 turns/일 시 ~$0.1.

옵션 B — Inline emit (assistant 응답에 piggy-back) 권장
// system prompt 추가:
"매 final answer 끝에 다음 형식으로 제목 갱신을 출력하세요:
<title>[10-20자 신규 제목]</title>
- 기존 제목과 이번 답변을 종합
- 대화의 누적 흐름 반영
- 토픽 크게 바뀌면 [checkpoint-suggested] 도 함께"

// renderer: stream 에서 <title>X</title> 추출 → session.title 갱신

장점: 비용 0 (이미 final answer 생성 중), 토픽 판단도 같이 가능

단점: LLM 따라 형식 일관성 (특히 weak model). fallback 로직 필요.

권장: B + A fallback
  • System prompt 에 `<title>` emit 지시 (B)
  • 매 turn 끝 stream 에서 `<title>X</title>` 추출 시도
  • 없거나 형식 깨졌으면 → A 로 fallback (mini-call)
  • 둘 다 fail 시 기존 title 유지 (안전)
  • title 길이 정규화 — strip + slice(0, 20)
Title 진화 예시
turn 1: 사용자 "오늘 회의 정리해줘"
        → title: "오늘 회의 정리"

turn 2: 사용자 "다음주 일정도 확인"
        → title: "회의 정리 + 다음주 일정 확인"

turn 3: 사용자 "PR 리뷰 진행하자"  ← 토픽 전환
        → assistant final 끝에 [checkpoint-suggested] emit
        → checkpoint 발생 → summary 저장
        → 새 session 시작 → title: "PR 리뷰 진행"

6. LLM context architecture (token 효율 + 맥락 보존)

매 LLM call 의 input 구조
┌────────────────────────────────────────────────┐
│ [system prompt]                              │ ← LVIS 본 동작 지시 + tools
├────────────────────────────────────────────────┤
│ [rolling summary preamble]                  │ ← 누적 압축 (max 2000 토큰)
│  - 어제까지 한 일 요약                          │
│  - 가장 최근 체크포인트 directives              │
├────────────────────────────────────────────────┤
│ [recent N turns full]                       │ ← 가장 최근 cushion (≥ 40k 토큰)
│  - user / assistant raw messages              │
├────────────────────────────────────────────────┤
│ [current user input]                        │ ← 이번 turn
└────────────────────────────────────────────────┘
Rolling summary 진화

매 체크포인트마다 incremental summary. 누적 summary 가 너무 두꺼워지면 자기 자신을 압축 (lossy):

checkpoint 1: "회의 정리 + 액션아이템 5건"
checkpoint 2: "회의 정리 + 액션 5건. PR 리뷰 진행"
checkpoint 3: (1+2 압축) → "회의 + PR 리뷰 (액션 5건)"
checkpoint 4: "회의/PR 리뷰. plugin grid v3 디자인"
...
Token 보존 전략 (OpenCode 패턴)
  • recent cushion: 최근 N turns (~40k tokens) raw 보존
  • summary preamble: max 2000 tokens (압축)
  • Hard limit: model context의 85% (15% 응답 여유)
  • Tier-1 trigger 도달 → cushion 의 가장 오래된 부분을 summary 로 이동
  • summary 가 2000+ 되면 자체 압축
