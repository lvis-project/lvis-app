Permission Review Scenario Board

English contract board for the permission UX. Builtin, plugin, and MCP tools enter the same ToolRegistry -> ToolExecutor -> PermissionManager path; source is display and audit metadata, not a separate permission lane.

S1Default workspace read

Read inside the workspace is allowed without approval and still leaves audit context.

S2Strict read asks

Strict mode asks for reads without invoking the reviewer.

S3Out-of-directory read

Reads outside the allowed directory switch to directory-scope approval.

S4Auto-review LOW mutation

Low-risk foreground mutation can run inline with reviewer and audit disclosure.

S5Auto-review MED mutation

Medium foreground mutation routes to the approval modal; the user remains the authority.

S6Auto-review HIGH shell

High-risk shell or deletion routes to approval and deny-once blocks execution.

S7Reviewer disabled

Modes without the LLM reviewer ask directly with clearer tool arguments.

S8Reviewer missing

Missing provider/model/key never silently allows; foreground asks and headless defers or denies.

S9Reviewer timeout/error

Reviewer failure is fail-closed HIGH and routes through the approval modal.

S10Category-less plugin tool

Invalid tool definitions fail closed at the host baseline and cannot hide behind user approval.

S11Overlay prompt import

overlay:... uses prompt import approval, then normal tool permission. The overlay lane stays separate from tool permission.

S12MCP remote tool

MCP tools use the same category policy with MCP source and low-trust floor.

S13Trusted metadata lookup

Trusted read-only metadata lookup can pass auto-review LOW where policy permits.

S14External content send

Sending summaries, files, meeting notes, or user data explains destination and payload before approval.

S15Unknown or unsafe network

Unknown host, non-HTTPS, broad token scope, or sensitive upload risk stays HIGH.

S16Headless network send

Headless routines defer non-LOW network sends and expose them through the manual queue button.
