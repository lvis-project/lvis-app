LVIS 질문 카드 (Ask-User-Question Card) — 디자인 목업

ask_user_question LLM 도구가 발생시키는 인라인 질문 카드의 디자인 의도 기록. 구현체는 src/ui/renderer/components/AskUserQuestionCard.tsx 에 존재하며, 채팅 스트림 끝에 인라인으로 렌더된다 (PR #334 의 floating panel 은 폐기되고 인라인으로 통합됨).

1. 디자인 의도 요약

질문 개수 레이아웃 응답 흐름 건너뛰기
1개 단일 폼: 질문 텍스트 + 선택지 칩 + 선택적 자유 입력 선택지 클릭 시 즉시 respond. 자유 입력은 "보내기" 버튼으로 전송. "건너뛰기" → dismissed: true 로 LLM 에 전달, LLM 은 기본값으로 fallback.
2~4개 페이지네이션 방식. 헤더에 n / total 표기.
마지막 단계 = 검토 화면 (모든 답변 나열, 행 클릭 시 해당 질문으로 복귀). 		다음/이전 → 마지막 단계 → 검토 → 보내기 로 모든 답변 일괄 전송. 어느 단계에서든 "건너뛰기" 시 카드 전체가 닫힘.

2. 시나리오 A — 단일 질문 (1/1)

가장 흔한 경로. LLM 이 3개 선택지를 emit, recommendedIndex: 1 · altIndices: [2] 로 표기. 어느 선택지든 클릭하면 즉시 응답하고 카드가 resolved 상태로 전환된다. 자유 입력은 allowFreeText: true 일 때만 노출.

"이번 주 회의록 정리 시작할까요?"
오후 2:14
회의록 정리 어떤 형식으로 만들어 드릴까요?
❓ 질문
출력 형식을 골라주세요. 회의 길이가 30분 이상이면 요약형이 더 보기 쉽습니다.
상태 표기 규칙
  • Recommend — 질문당 1개. recommendedIndex 가 둘 이상이어도 첫 번째만 사용.
  • 대안 — 보조 추천. Recommend 와 disjoint, 회색 톤.
  • 선택된 칩 = 파란색 배경. 같은 라벨이 중복돼도 인덱스로 구분 (label 비교 시 동시 활성화 버그가 있어 인덱스 비교로 변경됨).
  • 답변/선택지 길이 제한 없음 — UI 는 입력값을 그대로 보존하고 줄바꿈/wrap 으로 처리.

3. 시나리오 B — 다중 질문 (3개) 페이지네이션

각 질문을 단계별로 진행. 우측 하단 버튼이 다음 → 검토 (마지막 단계) → 보내기 (검토 화면) 로 순환. 이전 으로 복귀 가능. 카드의 step + drafts 상태는 request.id 별로 보존된다.

검토 행 라벨: 모델이 summaryHint 를 제공한 경우 그 값 (한국어 ≤ 10자 권고). 누락 시 question 을 약 14자에서 잘라서 사용 (truncateForLabel). 미응답 행은 답변 텍스트가 amber 색으로 표시.

4. 시나리오 C — 답변 완료 후 채팅 노출

핵심 변경점: 답변 후 카드만 사라지면 무엇을 답했는지 가 채팅 history 에 남지 않아 사용자가 회상하기 어렵다. 답변 시점에 카드를 resolved 상태로 어둡게 비활성화하고, 그 아래에 한 줄짜리 답변 회고 칩 을 user-bubble 우측 정렬로 추가해서 향후 스크롤 / 세션 reload 후에도 응답 내용이 보존되도록 한다.

구현 메모:
  • 현재 구현은 onResolved(request.id) 시점에 카드를 askQuestions 큐에서 제거 → DOM 에서 사라짐. 답변 회고 칩 미존재 = 본 시나리오 추가 분.
  • 답변 회고 칩은 kind: "ask_user_answer" (또는 system 변형) 새 ChatEntry 종류로 entries 에 push 하면 자연스러움. 채팅 history JSON 에 저장되어 새로고침 / 세션 reload 후에도 유지.
  • Resolved 카드는 entries 에 그대로 두되 시각적으로 비활성화. 이전 페이지로 스크롤하면 사용자가 어떤 질문에 어떻게 답했는지 그대로 확인 가능.
  • 다중 질문은 respondAskUserQuestion 한 번 호출 → 답변 칩 N개 push.

5. IPC + 상태 컨트랙트

이벤트 페이로드 UI 동작
ask_user_question 도구 emit {id, questions[], createdAt} ChatView 의 askQuestions 큐 끝에 카드 마운트.
단일 질문에서 선택지 클릭 respondAskUserQuestion({requestId, answers: [{choice}]}) 카드 resolved → 답변 회고 칩 push (시나리오 D 적용 시).
다중 질문 보내기 respondAskUserQuestion({requestId, answers: [{choice or freeText}, ...]}) 한 번의 IPC 호출로 모든 답변 일괄 전송. 답변 칩 N개 push.
건너뛰기 respondAskUserQuestion({requestId, dismissed: true}) LLM tool_result dismissed: true → 기본값 fallback. dismissed 마커 push.

6. 미해결 디자인 메모

  • 한 줄 입력 고정. 기존 popup 은 Textarea 라 세로로 비대했음 — 카드가 modal 처럼 느껴짐. 자유 입력은 단일 라인 input 으로만 받고, 길이 제한 없이 wrap 처리.
  • label 자체에 뱃지 baked-in 금지. 모델은 평문 답변만 emit, Recommend / 대안 표기는 recommendedIndex · altIndices 로 시각만 분리. 길이 제한이 없으므로 label 의 답변 텍스트는 자연스러운 길이로 emit 가능.
  • 인덱스 기반 선택 비교. 같은 라벨 중복 시 string 비교로 모두 활성화되는 버그가 있어 choiceIndex 로 변경.
  • request.id 별 상태 보존. 같은 id 로 재마운트되면 step + drafts 보존, 새 id 면 초기화.
  • 인라인 배치. 카드는 chat 스트림 끝 (활성 turn 의 entries 직후) 에 위치. PR #334 의 FloatingQuestionPanel 은 폐기 — 질문 흐름을 layered modal 이 아니라 conversation history 의 일부로 유지하기 위함.
  • 답변 회고 칩 (시나리오 D): 미구현. 새 entry kind 추가 필요. 우선순위 medium — 사용자가 무엇을 답했는지 기억 못 하는 UX 갭 해소용.
